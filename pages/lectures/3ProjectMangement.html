Personas: Understanding Your Users

Personas are 'imagined users' - character portraits of user types who might use your product.

Why Use Personas?

  • Help developers empathize with users
  • Allow team to "step into the user's shoes"
  • Prevent adding unnecessary features
  • Avoid assumptions based on developer knowledge
  • Create shared understanding within the team
Key Point: Personas should be based on real user research, not just assumptions!

Example Persona

Sarah Chen - Busy Professional

Age: 32 | Location: Oklahoma City | Role: Marketing Manager

Background:
MBA from OCU, 8 years experience
Tech Skills:
Intermediate, uses many SaaS tools
Goals:
Streamline team collaboration
Pain Points:
Too many disconnected tools

Quote: "I need one place to manage all our projects without switching between five different apps."

Creating Effective Personas

Full Personas (Based on User Research)

  • Study and survey potential users
  • Understand their jobs, background, aspirations
  • Abstract essential information about user types
  • Create detailed character portraits

Proto-Personas (Limited Information)

  • Created as team exercise
  • Based on available information
  • Better than no personas at all
  • Should be refined with real data later

For a dentist appointment system, you might create: Dentist persona, Receptionist persona, Patient persona

Scenarios: Stories of System Use

A scenario is a narrative describing how users interact with your product to achieve their goals.

Example Scenario: Sarah's Team Meeting

Sarah needs to schedule a team meeting for next week. She opens the app on her phone during her commute and checks team availability. She sees that Tuesday afternoon works for everyone. She creates the meeting, adds the agenda items from last week's action points, and assigns pre-meeting tasks to team members. Each person receives a notification with their specific preparation requirements. The system automatically adds the meeting to everyone's calendar and sends a reminder the day before.

Remember: Scenarios are NOT specifications - they're communication tools!

Writing Effective Scenarios

‚úÖ Good Practices

  • Write from user's perspective
  • Based on personas
  • Focus on user goals
  • Keep general, avoid implementation details
  • 2-3 paragraphs to 1 page

‚ùå Common Mistakes

  • Too technical
  • System-centric view
  • Too detailed
  • Including database schemas
  • Focusing on UI specifics

Create several scenarios for each persona to ensure coverage of all user roles

User Stories: Structured Needs

User stories are fine-grain narratives that describe specific things users want from the system.

Standard Format:

As a <role>, I want <feature> so that <benefit>

As a teacher, I want to notify all students when new materials are available so that everyone stays updated on course content.
As a receptionist, I need to view all appointments for the day so that I can prepare patient files in advance.
As an author, I want to organize my book into chapters and sections so that readers can navigate easily.

Identifying Features

From User Actions to System Features

Process:

  1. Highlight user actions in scenarios and stories
  2. Identify required functionality to support actions
  3. Group related functions into features
  4. Define feature boundaries and interactions

Scenario Action: "She checks team availability"

Required Feature: Availability Calendar with team view

Related Functions: View schedules, block time, set recurring availability

What Makes a Good Feature?

A software product feature is:

"A fragment of functionality that implements something a user may need or want"

‚úÖ Well-Defined Features

  • User-focused purpose
  • Clear boundaries
  • Testable outcomes
  • Independent value

‚ùå Poor Features

  • Technical implementation
  • Overlapping functionality
  • No clear user benefit
  • Too broad or too narrow

Managing Agile Development

Tools and Methodologies

Scrum Framework

  • Product Backlog: Prioritized list of user stories
  • Sprint Planning: Select stories for sprint
  • Daily Standups: Team synchronization
  • Sprint Review: Demo completed features
  • Retrospective: Team improvement

Kanban Method

  • Visual Board: See work status at a glance
  • WIP Limits: Prevent overload
  • Continuous Flow: No fixed iterations

Modern Deployment for Agile Teams

Cloud platforms enable rapid deployment and scaling of features.

Benefits for Agile Development:

  • Continuous Deployment: Ship features as they're ready
  • A/B Testing: Test features with user segments
  • Feature Flags: Enable/disable features remotely
  • Auto-scaling: Handle varying user loads
  • Global Reach: Deploy close to users

Microservices Architecture

Aligning Services with Features

Each major feature can be implemented as an independent microservice.

User Auth Service

Login, Registration, Password

Calendar Service

Scheduling, Availability

Notification Service

Email, SMS, Push

Benefit: Teams can develop and deploy features independently!

Case Study: Spotify's Agile Model

Spotify's Approach:

  • Squads: Small, cross-functional teams (like mini-startups)
  • Tribes: Collections of squads working on related features
  • Chapters: People with similar skills across squads
  • Guilds: Communities of interest

Example Squad: "Radio Experience"

Owns all features related to radio functionality. Creates personas for different music listeners, writes scenarios about discovering new music, and develops features like personalized stations and skip limits.

Key Insight: Each squad owns their features from concept to deployment!
