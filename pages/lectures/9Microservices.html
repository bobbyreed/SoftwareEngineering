What are Microservices?

Definition: A microservice is an independent and self-contained software component that runs in its own process and communicates with other microservices using lightweight protocols.
  • Small-scale, stateless services with single responsibility
  • Completely independent with own database and UI code
  • Can use different programming languages and databases
  • Combined to create applications

Problems with Monolithic Applications

Challenge 1: The whole system must be rebuilt, re-tested and re-deployed when any change is made

Challenge 2: As demand increases, the whole system must be scaled, even if demand is localized to specific components

Microservices architecture addresses both of these fundamental problems!

Benefits of Microservices Architecture

  • ‚úÖ Independent Deployment: Services can be stopped and restarted without affecting others
  • ‚úÖ Scalability: Service replicas can be quickly created and deployed
  • ‚úÖ Cost-Effective: 'Scaling-out' is cheaper than 'scaling up'
  • ‚úÖ Technology Flexibility: Different languages and databases per service
  • ‚úÖ Team Autonomy: Teams own their services end-to-end

Data Management Challenges

ACID transactions are impractical in microservices architecture

Two Types of Inconsistency:

  1. Dependent Data Inconsistency: Actions of one service cause another's data to become inconsistent
  2. Replica Inconsistency: Multiple replicas have different versions of the same data

Eventual Consistency

Databases need not be completely consistent all the time

Strategy: Eventually Consistent Systems

  • Accept temporary inconsistency
  • Implement reconciliation mechanisms
  • Use event sourcing for audit trail
  • Apply compensating transactions when needed

Failure Types in Microservices

Internal Service Failure

Detected by service, reported in error message (e.g., invalid URL)

External Service Failure

External cause affects availability, service becomes unresponsive

Service Performance Failure

Performance degrades to unacceptable level due to load or internal issues

Failure Management Strategies

  • üîÑ Retry Logic: Automatic retry with exponential backoff
  • ‚ö° Circuit Breaker: Stop calling failing services
  • üîÄ Fallback: Provide degraded functionality
  • ‚è±Ô∏è Timeouts: Don't wait indefinitely
  • üíì Health Checks: Monitor service status

Containers and Microservices

Continuous Deployment

Definition: As soon as a change to a service has been made and validated, the modified service is redeployed

Process is completely automated:

  • Automated testing checks production quality
  • Multiple versions may be maintained
  • Quick rollback if problems discovered
