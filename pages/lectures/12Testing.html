What is Software Testing?

Software testing is a process where you:

  • Execute your program using data that simulates user inputs
  • Observe its behavior to verify expected functionality
  • Find bugs and show the program meets requirements
Key Point: Testing can show the presence of bugs, but never their absence!

Unit Testing Fundamentals

A code unit is anything with a clearly defined responsibility

Core Principle:

If a program unit behaves as expected for a set of inputs with shared characteristics, it will behave the same way for a larger set with those characteristics

This allows us to test efficiently using representative inputs!

Equivalence Partitions

Divide all possible inputs into groups with shared characteristics

Test at least one input from each partition!

Example: Name Validation Function

Consider a function that validates names with these rules:

  • 2-40 characters long
  • Only alphabetic characters, hyphens, or apostrophes
  • Cannot start with hyphen or apostrophe
  • No double hyphens or quoted text
Challenge: How many test cases do we need?

Equivalence Partitions: Name Validator

Partition Description Example
Correct 1 Only alphabetic, 2-40 chars "John", "Mary"
Correct 2 With hyphens/apostrophes "O'Brien", "Smith-Jones"
Incorrect 1 Invalid characters "John123", "Mary!"
Incorrect 2 Wrong length "J", "A"*41
Incorrect 3 Starts with hyphen/apostrophe "-John", "'Mary"
Incorrect 4 Double hyphen or quotes "John--Doe", "O\"Brien\""

Boundary Testing Guidelines

Test at the Edges! üéØ

Bugs often hide at boundary conditions

  • Test edge cases: Min/max values, empty inputs
  • Force errors: Invalid inputs, error conditions
  • Fill buffers: Test overflow conditions
  • Repeat yourself: Same input multiple times

Test Automation

Tests should be executable!

An automated test includes:

  • Input data to the unit being tested
  • Expected result
  • Verification that actual equals expected

Build hundreds or thousands of executable tests!

The AAA Pattern

Structure automated tests with three parts:

1. Arrange

Set up test parameters and mock objects

2. Act

Call the unit being tested

3. Assert

Verify the expected outcome

AAA Pattern Example

function testValidateName() {
    // Arrange
    const validator = new NameValidator();
    const validName = "John-Smith";
    
    // Act
    const result = validator.validate(validName);
    
    // Assert
    assertEqual(result, true);
}

Clean, readable, and maintainable test structure!

Test-Driven Development (TDD)

Write tests BEFORE writing code!

The TDD Cycle:

  1. üî¥ Write a failing test
  2. üü¢ Write minimal code to pass
  3. üîµ Refactor the code

Introduced by Extreme Programming, works with any incremental approach

TDD Process Steps

Step Action
1. Identify partial implementation Break down functionality into mini-units
2. Write mini-unit tests Create tests for chosen mini-unit
3. Write code stub Incomplete code that will fail
4. Run all tests Previous pass, new test fails
5. Implement functionality Write code to pass the test
6. Refactor if needed Improve code while tests pass

Benefits of Test-Driven Development

  • ‚úÖ Systematic approach: Tests clearly linked to code
  • ‚úÖ Complete coverage: All code has tests
  • ‚úÖ Living documentation: Tests explain what code does
  • ‚úÖ Simplified debugging: Link failures to recent changes
  • ‚úÖ Simpler code: Only write what's needed to pass tests
