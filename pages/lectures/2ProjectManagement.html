Fragments of product functionality

Scenarios 📖

A narrative that describes how a user, or group of users, might use your system.

Key Characteristics:

  • Written from the user's perspective
  • Based on identified personas or real users
  • High-level stories of system use
  • 2-3 paragraphs to a page in length
  • Don't include implementation details
Remember: Scenarios help you understand how your software might be used - they're "tools for thinking"

User Stories ✍️

Finer-grain narratives that set out a single thing a user wants from the software

As a <role>, I <want|need> to <do something>
Example 1: As a teacher, I want to tell all members of my group when new information is available
Example 2: As an author, I need a way to organize the book that I'm writing into chapters and sections
As a <role>, I <want|need> to <do something> so that <reason>
Example 3: As a teacher, I need to report who is attending a class trip so that the school maintains required health and safety records

Identifying Features

Features can be identified from scenarios and stories

✅ Sources of Features

  • User Knowledge: What users want and need
  • Product Knowledge: Experience from existing products
  • Domain Knowledge: Understanding the work area
  • Technology Knowledge: Leveraging new tech capabilities

🔍 How to Identify

  • Highlight user actions in narratives
  • Look for active verbs (use, choose, create)
  • Think about supporting features needed
  • Consider innovative alternatives

Microservices Architecture

What are Microservices?

Small-scale, stateless services with a single responsibility, combined to create applications

✅ Advantages

  • Independent deployment
  • Scalable components
  • Technology diversity
  • Fault isolation

⚠️ Challenges

  • Complex communication
  • Distributed system issues
  • Testing complexity
  • Operational overhead

Introduction to Scrum 🏃‍♂️

An agile framework for managing product development

📋 Product Backlog

To-do list of features and improvements

⏱️ Timeboxed Sprints

Fixed 2-4 week development periods

👥 Self-organizing Teams

Teams make their own decisions

Sprint Goal: Deliver a potentially shippable product increment

The Scrum Cycle

1️⃣ Review product backlog

2️⃣ Select items to implement

3️⃣ Plan sprint → Create sprint backlog

4️⃣ Daily Scrum meetings

5️⃣ Develop software

6️⃣ Test software

7️⃣ Review sprint → Shippable increment

🔄 Repeat!

Group 1: Scrum Definition & Theory

Key Points to Cover:

  • Definition: Framework for developing and sustaining complex products
  • Empirical Process Control: Transparency, Inspection, Adaptation
  • Iterative & Incremental: Build in small pieces
  • Fixed-length iterations: 2-4 week sprints
  • Potentially shippable increments: Working software each sprint

Group 2: Scrum Team

👤 Product Owner

Manages product backlog

Ensures team focuses on value

🎯 ScrumMaster

Coaches the team

Removes impediments

👨‍💻 Development Team

5-8 cross-functional members

Self-organizing

Remember: No traditional project manager - the team self-organizes!

Group 3: Scrum Events

🎯 Sprint Planning

Select items from product backlog, create sprint backlog (≤1 day)

☀️ Daily Scrum

15-minute stand-up: What did I do? What will I do? Any impediments?

🚀 Sprint Execution

Team works to complete sprint backlog items

🔍 Sprint Review

Demo working software, get feedback

🔄 Sprint Retrospective

Team reflects on process improvements

Group 4: Scrum Artifacts

📋 Product Backlog

  • Prioritized list of features
  • Owned by Product Owner
  • Constantly refined
  • Contains user stories/epics

📝 Sprint Backlog

  • Selected items for sprint
  • Owned by Development Team
  • Updated daily
  • More detailed tasks

📦 Product Increment

Sum of all completed backlog items - potentially shippable product!
