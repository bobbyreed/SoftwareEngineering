Step 4: GitHub Actions Workflow

deploy.yml Configuration

  push:

Continue deploy.yml - Testing

    - name: Install dependencies

Step 6: Add Deployment Stage

Complete deploy.yml - Deployment

    - name: Build site

    - name: Deploy to GitHub Pages
      if: github.ref == 'refs/heads/main'
      uses: peaceiris/actions-gh-pages@v3

        publish_dir: ./

‚ö†Ô∏è IMPORTANT: Enable Workflow Permissions

Do this BEFORE pushing your code!

Without this step, your Action will fail with a 403 error.

Enable Write Permissions for GitHub Actions:

  1. Go to your repository on GitHub
  2. Click Settings (top menu bar)
  3. In the left sidebar, click Actions ‚Üí General
  4. Scroll down to "Workflow permissions" section
  5. Select "Read and write permissions"

    (Not "Read repository contents and packages permissions")

  6. ‚úÖ Check "Allow GitHub Actions to create and approve pull requests"
  7. Click Save at the bottom
Why? The deployment action needs permission to push to the gh-pages branch. By default, GitHub Actions only has read access for security reasons.

Key Concepts You Just Learned üéì

YAML = Yet Another Markup Language

A simple way to write configuration files. Indentation matters! (Like Python)

The Workflow Lifecycle

  1. Trigger: Something happens (you push code)
  2. Spin Up: GitHub creates a fresh computer for you
  3. Execute: Runs all your steps in order
  4. Report: Tells you if it worked or failed
  5. Clean Up: Destroys the computer (you don't pay for it!)
Pro Tip: Every time your workflow runs, it starts with a completely fresh computer. Nothing from previous runs carries over!

Understanding the YAML

Key Components Explained

  • on: Triggers (push, pull_request, schedule)
  • jobs: Parallel or sequential tasks
  • runs-on: Runner environment (ubuntu, windows, macos)
  • steps: Individual actions in sequence
  • uses: Pre-built actions from marketplace
  • run: Shell commands to execute
  • if: Conditional execution logic
  • secrets: Secure credential storage
