What is the Cloud? ☁️

Definition: A large number of remote servers offered for rent by companies that own these servers.
  • Access servers remotely over the internet
  • Pay for the amount of server time used
  • Servers can be started and shut down on demand
  • 'Virtual servers' implemented in software

Traditional vs Cloud Computing

❌ Traditional Infrastructure

  • High upfront costs
  • Physical hardware management
  • Fixed capacity
  • Long procurement times
  • Maintenance overhead

✅ Cloud Infrastructure

  • Pay-as-you-go model
  • No hardware management
  • Elastic scaling
  • Instant provisioning
  • Managed services

Benefits of Cloud for Development

💰 Cost Efficiency

Avoid initial capital costs of hardware procurement

⚡ Quick Startup

Servers up and running in minutes, not weeks

🔄 Flexible Resources

Upgrade or downgrade servers as needed

🌍 Distributed Teams

Same environment accessible from anywhere

Virtual Servers 💻

Key Concept: A virtual server is a stand-alone system that can run on any hardware in the cloud

  • Made up of OS + software packages
  • 'Run anywhere' characteristic
  • No external dependencies
  • Implemented using Virtual Machines (VMs)

Virtualization Technology

Definition: Technology that allows multiple server instances to run on the same physical computer

How it works:

  • 🖥️ Hypervisor: Provides hardware emulation
  • 🔧 Isolation: Each VM is completely separate
  • 🎯 Flexibility: Run different OS on same hardware
  • 📦 Portability: Move VMs between physical servers

Virtual Machines vs Containers

🖥️ Virtual Machines

  • Full OS included
  • Gigabytes in size
  • Minutes to start
  • Complete isolation
  • Hardware virtualization

📦 Containers

  • Share host OS
  • Megabytes in size
  • Seconds to start
  • Process isolation
  • OS virtualization

Why Use Containers?

Container Advantages

  • ✅ Solve dependency problems
  • ✅ Ship complete environments
  • ✅ Enable portability across clouds
  • ✅ Support microservices architecture
  • ✅ Simplify DevOps adoption

Docker: Container Management 🐳

Docker: The most widely used container technology, developed by Google (2007), mainstream since 2015

Docker Components:

  • 📝 Dockerfile: Defines the container
  • 🖼️ Images: Built from Dockerfiles
  • 📦 Containers: Running instances
  • 🏪 Docker Hub: Registry for sharing

Docker System Architecture

Docker Client → Docker Host → Registries
     ↓              ↓             ↓
  Commands    Docker Daemon   Docker Hub
                    ↓
              Images & Containers
Workflow: Define (Dockerfile) → Build (Image) → Run (Container) → Share (Registry)

Docker in Practice

Sample Dockerfile:

FROM node:14-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["npm", "start"]
Pro Tip: Each line creates a new layer - optimize for caching!

  • ⚡ Auto-scaling capabilities

Heroku, Google App Engine, Azure App Service

  • 🌐 Web-based access
  • 🔄 Automatic updates
  • 📱 Multi-device support

Examples:

Salesforce, Office 365, Google Workspace, Dropbox

Key Architecture Decision: Database

Multi-tenant vs Multi-instance

🏢 Multi-tenant

All customers share the same database

🏠 Multi-instance

Each customer has their own database

Multi-tenant Database Systems

✅ Advantages

  • Resource efficiency
  • Simplified management
  • Easier updates
  • Lower costs

❌ Disadvantages

  • Schema inflexibility
  • Security concerns
  • Increased complexity
  • Performance isolation

Multi-instance Database Systems

✅ Advantages

  • Complete flexibility
  • Strong security
  • Custom scaling
  • Failure isolation

❌ Disadvantages

  • Higher costs
  • Complex updates
  • Resource overhead
  • Management burden

Choosing Database Architecture

Key Questions to Ask:
  • 🎯 Do customers need schema customization?
  • 🔒 Are there strict security/compliance requirements?
  • 💾 What's the typical database size?
  • 🔄 Do you need ACID transactions?
  • 🏗️ Are you using microservices?
  • 🔌 Need to integrate existing databases?

Scalability in the Cloud 📈

Scalability: The ability to adapt automatically to changes in system load

⬆️ Scaling Up

Increase server power (vertical scaling)

  • More CPU/RAM
  • Limited by hardware
  • Requires downtime

➡️ Scaling Out

Add more servers (horizontal scaling)

  • Distributed load
  • Virtually unlimited
  • No downtime

Auto-scaling in Practice

# Auto-scaling Policy Example
Metric: CPU Utilization
Scale Out: When CPU > 70% for 5 minutes
  → Add 2 instances
Scale In: When CPU < 30% for 10 minutes
  → Remove 1 instance
Min Instances: 2
Max Instances: 10
Best Practice: Use multiple metrics (CPU, memory, request rate) for scaling decisions

Building Resilient Systems 🛡️

Resilience: Ability to continue delivering critical services during failures

Key Strategies:

  • 🔄 Redundancy and replication
  • ⚡ Fast restart capabilities
  • 🌍 Geographic distribution
  • 📊 Health monitoring
  • 🔀 Automatic failover

Active-Standby Pattern

Location A              Location B
    ↓                      ↓
Active System ← Monitor → Standby System
    ↓                      ↓
Database 1 ← Mirroring → Database 2
How it works: If active system fails, standby takes over immediately

Cloud & Microservices 🔗

Perfect Match: Microservices architecture is ideal for cloud deployment

Why they work well together:

  • 📦 Each service in its own container
  • 🎯 Independent scaling per service
  • 🔄 Easy updates and rollbacks
  • 💡 Fault isolation
  • 🚀 Continuous deployment friendly

Choosing a Cloud Platform

No 'Best' Platform - It Depends!

Major Players:

  • 🟠 AWS: Most comprehensive, market leader
  • 🔵 Azure: Enterprise-friendly, Microsoft integration
  • 🔴 Google Cloud: Strong in AI/ML, Kubernetes
  • 🟣 Specialized: SAP Cloud, Salesforce, etc.

Technical Platform Considerations

Evaluate Based On:

  • 📊 Expected load and predictability
  • 🛡️ Resilience requirements
  • 🔧 Supported services and tools
  • 🔒 Privacy and data protection
  • 🌍 Geographic coverage
  • ⚡ Performance requirements

Business Platform Considerations

Business Factors:

  • 🎯 Target customer expectations
  • 💰 Cost structure and pricing
  • 📝 Service Level Agreements (SLAs)
  • 🔄 Portability and migration options
  • 👩‍💻 Developer experience and tooling
  • 📚 Documentation and support

Avoiding Vendor Lock-in 🔓

Risk: Becoming too dependent on one cloud provider's proprietary services

Mitigation Strategies:

  • 🐳 Use containers for portability
  • 🔧 Stick to standard technologies
  • 📚 Abstract cloud-specific APIs
  • 🌐 Consider multi-cloud strategy
  • 📦 Use cloud-agnostic tools (Kubernetes, Terraform)
