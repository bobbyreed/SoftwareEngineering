Peer Review Process >

How Peer Review Works

Before the presentation: You'll be assigned a partner

During the presentation: Your partner will:

  • Watch your presentation carefully
  • Explore your GitHub repository
  • Create 3 GitHub issues with constructive feedback

After the presentation: Your partner will:

  • Explain each of the 3 issues they created
  • Discuss why they're valuable improvements

Creating Good GitHub Issues =Ý

What Makes a Good Issue?

 DO

  • Be specific and actionable
  • Provide context and examples
  • Be constructive and helpful
  • Suggest improvements, not just problems
  • Include steps to reproduce (for bugs)

L DON'T

  • Be vague ("fix the code")
  • Be critical without being helpful
  • Create duplicate issues
  • Nitpick minor style preferences
  • Focus only on cosmetic changes

Types of Issues to Consider <÷

= Bugs

Issues that cause errors or unexpected behavior

Example: "Button click causes console error"

( Enhancements

New features or improvements to existing functionality

Example: "Add dark mode toggle"

=Ú Documentation

Missing or unclear README, comments, or setup instructions

Example: "Add installation steps to README"

= Security

Potential security vulnerabilities or best practices

Example: "API key exposed in client code"

Goal: Aim for a diverse set of issues that help improve different aspects of the project
