Part 2: Introduction to UML üìê

Unified Modeling Language

The Universal Language of Software Design

What is UML?

UML is a standardized visual language for modeling software systems. It helps us communicate design ideas, document systems, and plan implementations before writing code.

Today we'll cover:

  • Use Case Diagrams - What the system does
  • Class Diagrams - System structure
  • Sequence Diagrams - Interactions over time
  • Activity Diagrams - Process workflows

üìù Communication

Use Case Diagrams üé≠

What are Use Case Diagrams?

Use Case Diagrams show what a system does from the user's perspective. They capture functional requirements by showing actors (users) and their interactions with the system.

Key Components:

  • Actors: Users or external systems (stick figures)
  • Use Cases: System functions (ovals)
  • System Boundary: Box around use cases
  • Relationships: Lines connecting actors to use cases

When to use: Requirements gathering, understanding user needs, initial system design

Class Diagrams üèóÔ∏è

What are Class Diagrams?

Class Diagrams show the structure of a system by modeling classes, their attributes, methods, and relationships. This is the most commonly used UML diagram.

Class Structure:

  • Top: Class name
  • Middle: Attributes (properties)
  • Bottom: Methods (behaviors)

Relationships:

  • Association: Solid line
  • Inheritance: Arrow with hollow head
  • Composition: Filled diamond

When to use: Detailed system design, database design, object-oriented programming planning

Class Diagram Example: Library System üìö

Key Points: Librarian inherits from Member (reuses attributes). Loan connects Book and Member (association class).

Step-by-Step: Class Diagrams in draw.io

1Ô∏è‚É£ Enable UML Library

2Ô∏è‚É£ Add Class Box

3Ô∏è‚É£ Fill in Class Details

Sequence Diagrams ‚è±Ô∏è

What are Sequence Diagrams?

Sequence Diagrams show how objects interact over time. They capture the flow of messages between components in a specific scenario.

Components:

  • Actors/Objects: Boxes at top
  • Lifelines: Dashed vertical lines
  • Messages: Horizontal arrows
  • Activation: Thin rectangles on lifeline

Message Types:

  • Synchronous: Solid arrow
  • Return: Dashed arrow
  • Create: Arrow to new object

When to use: Understanding complex interactions, API design, debugging logic flows

Sequence Example: Borrow Book üìö

Reading tip: Time flows downward. Each arrow represents a method call or message between objects.

How to Create This in draw.io üíª

Step-by-Step: Sequence Diagrams in draw.io

1Ô∏è‚É£ Add Objects/Actors

  • From UML library: Drag "Object" boxes
  • Or use "Actor" for users
  • Arrange horizontally across the top
  • Double-click to add names

2Ô∏è‚É£ Add Lifelines

  • From UML library: Drag "Lifeline"
  • Or use dashed line from General shapes
  • Connect from bottom of each object, going down
  • Set line style to dashed

3Ô∏è‚É£ Add Messages

  • Use Arrow connector
  • Draw from left lifeline to right
  • Solid arrow = synchronous call
  • Double-click arrow to add message text

4Ô∏è‚É£ Add Return Messages

  • Draw arrow from right back to left
  • Change line style to dashed
  • Returns usually have shorter labels

5Ô∏è‚É£ Add Activation Boxes

  • Thin rectangles on the lifeline
  • Show when object is "active" or processing
  • Overlay on the dashed lifeline
  • Start where message arrives, end when done

6Ô∏è‚É£ Number the Messages

  • Add numbers: 1:, 2:, etc.
  • Helps show execution order
  • Place before the message name
Pro Tip: Use Arrange ‚Üí Insert ‚Üí Horizontal Space to add room between messages. Keep objects evenly spaced for clarity!

Activity Diagrams üîÑ

What are Activity Diagrams?

Activity Diagrams show workflows and business processes. They're like flowcharts on steroids, modeling the flow of control from activity to activity.

Components:

  • Start: Filled circle
  • Activity: Rounded rectangle
  • Decision: Diamond
  • End: Bull's eye (circle in circle)

Control Flow:

  • Fork: Split into parallel flows
  • Join: Merge parallel flows
  • Swimlanes: Group by responsibility

When to use: Business process modeling, workflow automation, algorithm visualization

Activity Example: Return Book üìö

Notice: Diamonds represent decisions. All paths eventually lead to the end node.

How to Create This in draw.io üíª

Step-by-Step: Activity Diagrams in draw.io

1Ô∏è‚É£ Add Start Node

  • From General shapes: Circle
  • Make it small and filled (solid black)
  • Place at the top of your diagram

2Ô∏è‚É£ Add Activities

  • Use Rounded Rectangle
  • From UML library: "Action" shape
  • Double-click to add activity name
  • Example: "Scan book barcode"

3Ô∏è‚É£ Add Decision Nodes

  • From General shapes: Diamond
  • Add question text inside (e.g., "Overdue?")
  • Decision nodes have 2+ outgoing arrows

4Ô∏è‚É£ Connect with Arrows

  • Use solid Arrow connectors
  • Connect in sequence: Start ‚Üí Activity ‚Üí Decision
  • Label decision branches: "Yes", "No"

5Ô∏è‚É£ Merge Paths

  • Multiple arrows can converge to one point
  • Use small circle or directly to next activity
  • All paths should eventually reach the end

6Ô∏è‚É£ Add End Node

  • Bull's eye symbol: Circle inside circle
  • Outer circle = hollow, inner = filled
  • Place at bottom where workflow ends
Swimlanes Tip: Want to show different actors? Use Table from General shapes with vertical columns. Place activities in appropriate columns!

UML Best Practices ‚≠ê

‚úÖ Do

  • Keep diagrams simple and focused
  • Use consistent naming conventions
  • Update diagrams when code changes
  • Choose the right diagram for the task

‚ùå Don't

  • Try to show everything in one diagram
  • Over-document simple systems
  • Treat diagrams as final implementation
  • Use UML for its own sake

The Golden Rule

UML is a communication tool, not a straightjacket.
Use it to clarify thinking and share ideas, not to document every detail.

"The code is the truth, but it's not the whole truth." - Grady Booch
