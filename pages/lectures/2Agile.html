Plan-Driven vs Agile Development

üìã Plan-Driven

  • Heavy documentation
  • Detailed upfront planning
  • Formal quality assurance
  • Good for large, critical systems
  • High overhead for small projects
  • Slow to respond to changes

üöÄ Agile

  • Rapid response to change

Five Core Agile Principles

1. üë• Involve the Customer

Customers work closely with the development team to provide requirements and evaluate each increment

2. üîÑ Embrace Change

Expect features and details to change as the team learns more. Adapt the software to cope with changes

3. üì¶ Develop and Deliver Incrementally

Always develop in increments. Test and evaluate each increment, feeding back required changes

4. üéØ Maintain Simplicity

Focus on simplicity in both software and process. Eliminate complexity wherever possible

5. üåü Focus on People, Not Process

Trust the team to develop their own ways of working without prescriptive processes

The Birth of Agile: Snowbird 2001 üèîÔ∏è

The Meeting That Changed Software Development

February 11-13, 2001 - Snowbird Ski Resort, Utah

The Problem: By the late 1990s, software developers were frustrated with heavyweight, documentation-driven methodologies that couldn't keep up with rapidly changing business needs.

The Gathering: 17 software developers‚Äîincluding Kent Beck (XP), Jeff Sutherland and Ken Schwaber (Scrum), and others‚Äîmet at a ski lodge to discuss lightweight development methods.

What happened: Over three days of discussion (and skiing!), they found common ground despite coming from different methodologies. They drafted the Agile Manifesto and its 12 principles.

The Impact: What started as a weekend gathering became a global movement. The word "Agile" was chosen because it captured their goal: the ability to respond quickly to change.

Fun Fact: The original manifesto was written on a whiteboard and later published at agilemanifesto.org, where you can still see all 17 original signatories today!

Writing Scenarios üìñ

A narrative describing how users might use your system

Key Characteristics:

  • Written from user's perspective
  • Based on personas or real users
  • High-level stories of system use
  • 2-3 paragraphs to a page long
  • No implementation details needed
Example Scenario: Emma is a teacher who needs to organize her class trip. She logs into the school system from home using her Google credentials, accesses the trip management app, and creates a new trip entry with student names, emergency contacts, and medical information...
Remember: Scenarios are "tools for thinking" - they help understand how software might be used

User Stories ‚úçÔ∏è

Structured descriptions of a single user need

As a <role>, I <want|need> to <do something>
Example 1: As a teacher, I want to notify all parents when new information is available
Example 2: As an author, I need to organize my book into chapters and sections
As a <role>, I <want|need> to <do something> so that <reason>
Example 3: As a teacher, I need to report trip attendance so that the school maintains required health and safety records
Scrum Note: Product backlogs are often represented as sets of user stories

Extreme Programming (XP) ‚ö°

Developed by Kent Beck in 1998 - pushing good practices to "extreme" levels

The XP Philosophy:

If something is good, do it to the extreme!

  • Testing is good? ‚Üí Write tests first!
  • Code reviews are good? ‚Üí Review continuously (pair programming)
  • Integration is good? ‚Üí Integrate several times per day
  • Customer feedback is good? ‚Üí Have customer on-site
Impact: XP changed software development culture and introduced practices now considered mainstream

XP Core Practices

üìù Incremental Planning / User Stories

No "grand plan" - requirements written as user stories, prioritized by business value

üöÄ Small Releases

Minimal useful functionality first, frequent releases adding incremental value

üß™ Test-Driven Development

Write tests BEFORE code - ensures always having tested, working version

üîÑ Continuous Integration

Integrate immediately after task completion - all tests must pass

üî® Refactoring

Constantly improve code structure, readability, efficiency, and security

Client-Server Architecture

Multiple Clients ‚Üí Load Balancer ‚Üí Server Farm

Client Side: User Interface (browser/mobile app)

Server Side: Business Logic + Database

Protocol: HTTP/HTTPS

Data Format: JSON or XML

Characteristics:

  • Suited for shared database applications
  • Functionality distributed between client and servers
  • Scalable through server replication
  • Load balancing for performance

Microservices Architecture

Definition:

Small-scale, stateless services with single responsibilities, combined to create applications

‚úÖ Benefits vs Monolithic

  • Individual service deployment
  • Selective scaling
  • Technology flexibility
  • Fault isolation
  • Team independence

‚ö†Ô∏è Challenges

  • Complex communication
  • Distributed system issues
  • Testing complexity
  • Service coordination
  • Operational overhead
Best For: Cloud-based products needing adaptability, scalability, and resilience
